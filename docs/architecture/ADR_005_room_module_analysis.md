# ADR 005: 房间模块架构设计分析

## 状态
已分析 / 架构合理

## 背景
用户请求分析“房间模块”的整体架构设计是否合理，包括数据库结构及依赖关系。
该模块是应用的核心交互单元，涉及创建、加入、游戏记录和结算。

## 架构概览
房间模块采用了 **"Serverless Real-time State Machine"** (无服务器实时状态机) 模式。
-   **状态 (State)**: 存储在 `rooms` 集合中（活跃玩家、当前分数）。
-   **事件 (Event)**: 存储在 `rounds` 集合中（每一局的得分）。
-   **流 (Stream)**: 前端通过 `db.watch` 实时订阅状态变化。

### 1. 数据库结构评价

#### A. `rooms` 集合 (状态快照)
```json
{
  "_id": "UUID",
  "roomId": "123456", // 6位短码，用于搜索
  "status": "active", // active | settled | finished
  "players": [
    { "id": "openid_A", "totalScore": 100, "hasLeft": false }
  ],
  "host": { ... },
  "clubId": "..."
}
```
*   **优点**:
    *   将 `rounds`（历史对局）从 `rooms` 文档中剥离，避免了 MongoDB 单文档 16MB 的限制，**设计非常合理**。
    *   包含 `players` 快照，使得“加入房间”和“显示当前分数”非常快，无需实时聚合。

#### B. `rounds` 集合 (事件日志)
```json
{
  "_id": "UUID",
  "roomDocId": "UUID",
  "scores": { "openid_A": 10 },
  "type": "game"
}
```
*   **优点**:
    *   作为不可变的“真理之源”，保证了计分的审计能力。
    *   支持 `aggregator` 在任何时候重放数据来重建统计。

### 2. 关键流程与依赖分析

#### A. 创建与加入 (Create/Join)
*   **机制**: 严格使用 `Transaction` (事务)。
*   **评价**:
    *   **合理性**: 高分。多人同时加入同一个房间是高并发场景，使用事务避免了“超员”和“数据覆盖”。
    *   **隐式依赖**: `joinRoom` 会检测 `clubId` 并自动将新用户加入圈子 (`club_members`)。这增加了 `joinRoom` 的职责，但对于简化用户体验（进房即进圈）是必要的权衡。
    *   **亮点**: 支持 "断线重连" (Re-join) 和 "散房积分恢复" (Score Recovery)，用户体验考虑周全。

#### B. 读写分离 (Read/Write)
*   **写**: 仅通过云函数 (`joinRoom`, `submitRound`, `settleRoom`) 进行，保证了规则校验（如：只有房主能结算，只有在房玩家能记分）。
*   **读**: 前端直接 `watch` `rooms` 文档。
*   **评价**: 这是微信小程序云开发的经典模式，既利用了云数据库的实时性，又保证了写入安全。

### 3. 潜在风险与改进建议

#### A. ID 双轨制 (Dual ID System)
*   **现象**: 系统中有 `roomId` (6位数字) 和 `_id` (32位 UUID)。前端和后端经常需要在两者间转换。
*   **风险**: 之前的统计 Bug 就是因为混淆了这两个 ID。
*   **建议**: 后端逻辑应统一下沉为使用 `_id` (UUID) 进行关联查询，`roomId` 仅作为用户可见的搜索入口。目前 `joinRoom` 已处理此逻辑，但需警惕新功能开发。

#### B. 幽灵房间 (Zombie Rooms)
*   **现象**: 如果房主创建房间后不结算直接消失，`active` 房间会一直残留。
*   **建议**: 建议增加 `autoSettle` (自动结算) 或 `TTL` (过期自动清理) 机制，基于 `updatedAt` 字段清理超过 24 小时未活动的房间。

## 结论

**房间模块的架构设计是合理且优秀的。**

1.  **数据模型成熟**: 采用了 Event Sourcing 的变体（`rooms` 存快照，`rounds` 存日志），既能在前端快速展示，又能保证数据可追溯。
2.  **并发控制严格**: 核心操作全事务化，保证了数据一致性。
3.  **扩展性强**: 将无限增长的 `rounds` 数据分离，使得单次对局即使打几百把也不会撑爆数据库。
